회원관리

회원 목록과 미국 기준 주소/연락처를 관리하고, 이메일 또는 챗 발송 기록을 남깁니다.

DOCUMENT: 클라이언트 회원가입/로그인 1차 운영 및 실제 배포 전환

현재 1차 구현은 빠른 검증을 위해 회원 데이터를 JSON 파일에 저장하고 Django 세션으로 로그인 상태를 유지한다. 비밀번호는 평문 저장 없이 Django 해시 값으로 보관한다.

  • 1차 범위: 회원가입, 로그인, 로그아웃, 현재 로그인 회원 조회, 관리자 회원목록 연동.
  • 회원가입 입력값: 아이디, 닉네임, 이메일, 비밀번호, 전화번호, 미국 기준 주소, 복수 기념일.
  • 관리자 확인: 클라이언트에서 가입한 회원은 이 회원관리 목록에 자동으로 추가된다.
  • 실제 배포 전 필수 전환: JSON 저장소를 SQLite/PostgreSQL 기반 DB와 Django Auth 또는 커스텀 User 모델로 이전한다.
  • 보안 검토: CSRF 보호, HTTPS, 세션 쿠키 보안 옵션, 비밀번호 정책, 로그인 실패 제한, 이메일 인증, 비밀번호 재설정 플로우를 적용한다.
  • 개인정보 검토: 주소, 전화번호, 기념일은 개인정보로 취급하고 보관기간, 삭제 요청, 관리자 접근 권한, 로그 마스킹 정책을 정한다.
  • 운영 검토: 약관/개인정보 동의, 마케팅 수신 동의, 이메일/챗 실제 발송 API, 관리자 권한 분리, 백업/복구 절차를 확정한다.
  • AI 상담 연동 전 검토: 회원 프로필과 기념일 정보를 프롬프트에 넣을 때 사용자 동의, 언어별 질문 유형, 민감정보 제외 규칙을 함께 적용한다.

기념일

이메일 / 챗 발송

이메일은 Hiworks SMTP로 실제 발송됩니다. 챗은 아직 외부 API 미연동 상태라 발송 로그만 저장합니다.

앱 페이지 관리

전체 홈페이지 타입과 마법사 구조에 대한 기획 문서입니다. 현재는 기능 구현이 아닌 운영 기준 문서로 제공합니다.

개요

이 방향은 AURA를 단순 페이지가 아니라 “페이지 타입/템플릿 기반 앱 빌더”로 확장하는 구조입니다. 현재 관리자 화면이 문구, API key, 사진, 스타일로 나뉘고 있으므로, 그 위에 앱 페이지 관리를 최상위 개념으로 두는 것이 자연스럽습니다.

관리 섹션 구성

앱 페이지 관리
- 현재 적용 타입: A1
- 타입 목록: A1, A2, B1, B2, B3
- 새 타입 생성
- 기존 타입 복제
- 타입 이름/설명 변경
- 타입 삭제
- 타입 적용
- 미리보기

페이지 타입의 의미

각 타입은 하나의 “홈페이지 설정 묶음”입니다.

A1 타입
- 문구 설정
- 사진 설정
- 스타일 설정
- 채팅/전문가 버튼 설정
- 섹션 구성
- 기본 언어
- 첫 화면 자동 전환 여부
- API 연결 방식

{
  "activePageType": "A1",
  "pageTypes": {
    "A1": {
      "name": "AURA 기본형",
      "description": "메이크업 중심 토탈 큐레이션",
      "content": {},
      "photos": {},
      "styles": {},
      "features": {}
    },
    "A2": {
      "name": "AURA 프리미엄형",
      "description": "전문가 상담 중심",
      "content": {},
      "photos": {},
      "styles": {},
      "features": {}
    }
  }
}

관리 흐름

  1. 앱 페이지 관리에서 A1 선택
  2. 문구관리로 가면 A1의 문구를 편집
  3. 사진관리로 가면 A1의 사진을 편집
  4. 스타일관리로 가면 A1의 스타일을 편집
  5. A2 선택 시 모든 관리 화면이 A2 기준으로 전환

타입 예시

  • A1: 메이크업 중심 앱
  • A2: 헤어/메이크업 혼합 앱
  • B1: 패션 에디토리얼 앱
  • B2: 인테리어 큐레이션 앱
  • B3: 전문가 상담 랜딩형 앱

구현 시 주의점

  1. 삭제는 바로 삭제하지 말고 비활성/보관 처리 추천
  2. A1 같은 기본 타입은 삭제 불가로 잠금 처리 추천
  3. 새 타입 생성보다 “A1 복제 후 A2로 수정” 흐름이 중요
  4. 편집 중인 타입과 실제 적용 타입은 분리 추천

API key 관리

AI 상담용 LLM을 복수 선택하고, provider별 API key와 LLM 버전을 관리합니다. 저장된 key는 전체 노출하지 않고 마스킹합니다.

추후 질문유형 관리 DB 설계

질문유형은 언어별로 분리해 관리할 예정입니다. LLM provider 설정과 독립된 구조로 두면, 같은 질문유형을 ChatGPT/Gemini/Claude 중 선택된 모델에 공통 적용할 수 있습니다.

questionTypes

  - id: makeup_consult

  - promptType: consultation
  - enabled: true
- en

  - promptType: consultation
  - enabled: true
